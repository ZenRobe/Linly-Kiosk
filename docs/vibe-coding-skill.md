# Vibe Coding 工作流

本 skill 定义了一套完整的功能开发流程，共 7 个阶段。严格遵循以下规则，在每个阶段执行对应的动作。

## 阶段总览

| Phase | 名称     | 核心动作                   | 产出物                            |
| ----- | -------- | -------------------------- | --------------------------------- |
| 0     | 初始化   | 恢复状态 / 创建分支        | feature 分支 + `.vibe/state.json` |
| 1     | 需求制定 | 探索代码库、讨论需求细节   | 中文需求文档（`.vibe/`）          |
| 2     | 执行计划 | 深度探索代码、制定实施步骤 | 分步执行计划                      |
| 3     | 编码     | 按计划逐步实现             | 可工作的代码                      |
| 4     | Review   | 质量检查、找 bug 和副作用  | 问题清单 + 修复                   |
| 5     | 文档     | 回顾 diff、写两份文档      | 技术参考 + 人类总结               |
| 6     | 提交     | 清理、提交、推送           | 远端分支，准备提 MR               |

## 状态持久化

使用 `.vibe/state.json` 保存工作流状态，确保上下文压缩或新会话后可恢复。

**结构**：

```json
{
  "feature": "功能简短名称",
  "branch": "feature/xxx",
  "phase": 0,
  "artifacts": {
    "requirements": ".vibe/xxx-requirements.md",
    "plan": ".vibe/xxx-plan.md",
    "summary": ".vibe/xxx-summary.md"
  }
}
```

**规则**：

- 每次阶段转换时更新 `phase` 字段
- 产出物生成后记录到 `artifacts`
- 新功能开始时重置整个 state

## 阶段检测

进入对话时：

1. 检查 `.vibe/state.json` 是否存在
   - **存在** → 读取，展示当前功能名、分支、阶段、已有产出物，请用户确认是否继续
   - **不存在** → 进入 Phase 0 初始化
2. 辅助验证：运行 `git branch --show-current` 确认分支与 state 记录一致
3. 如有冲突（比如分支不匹配），告知用户并请其决定

---

## Phase 0: 初始化

**进入条件**：用户要开始一个新功能，或无 `.vibe/state.json`。

**必做动作**：

### 新功能

1. 检查当前分支和 git 状态

2. 基于用户描述提议分支名（格式 `feature/xxx`），让用户确认

3. 用户确认后：切到主分支 → `git pull` → 创建新分支

4. 创建 `.vibe/` 目录（如不存在），写入初始 `state.json`：

   ```json
   { "feature": "xxx", "branch": "feature/xxx", "phase": 0, "artifacts": {} }
   ```

### 继续已有功能

1. 读取 `.vibe/state.json`，展示状态

2. 确认分支和阶段后直接进入对应 phase

   **关键规则**：

- **一个功能对应一个分支**，不要在同一分支做多个不相关功能
- 不要替用户决定阶段，只提供推测依据

---

## Phase 1: 需求制定

**进入条件**：分支已创建，开始明确需求。

**必做动作**：

1. 探索代码仓库结构，阅读已有文档，理解项目现状和已有功能

2. 与用户讨论需求细节，覆盖以下维度：

   - **功能需求**：做什么、为什么做、业务价值
   - **UI / 交互细节**：界面布局、操作流程、状态变化。主动提示用户是否参考项目中已有的 UI 模式
   - **边界情况**：错误处理、权限控制、兼容性

3. 将讨论结果整理为**中文需求文档**，存入 `.vibe/`（命名 `xxx-requirements.md`）

4. 更新 `state.json`：`phase → 1`，记录 `artifacts.requirements`

5. 用户确认需求文档后，提示：「需求已确认。建议输入 `/compact` 压缩上下文，然后说"进入 Phase 2"开始制定执行计划。压缩后我会从 state.json 恢复状态。」

   **产出物**：`.vibe/xxx-requirements.md`

---

## Phase 2: 执行计划

**进入条件**：需求文档已确认。

**必做动作**：

1. 读取需求文档（从 `state.json` 的 `artifacts.requirements` 获取路径）

2. 深度探索代码仓库：定位要修改的文件、理解现有架构和模式

3. 制定分步执行计划，每一步包含：

   - 具体要改的文件路径
   - 要添加/修改的函数名或模块
   - 改动内容的简要描述

4. 将计划存入 `.vibe/xxx-plan.md`

5. 更新 `state.json`：`phase → 2`，记录 `artifacts.plan`

6. 用户确认计划后，提示：「计划已确认。建议输入 `/compact` 压缩上下文，然后说"进入 Phase 3"开始编码。压缩后我会从 state.json 恢复状态。」

   **产出物**：`.vibe/xxx-plan.md`

---

## Phase 3: 编码

**进入条件**：执行计划已确认。

**必做动作**：

1. 从 `state.json` 读取计划路径，加载执行计划

2. 按计划逐步执行，每完成一步标记进度

3. 严格遵守项目的编码规范（参考 CLAUDE.md）

4. 遇到计划外的情况（需要额外改动、发现阻塞），主动告知用户并调整

5. 编码过程中定期做编译检查确保通过

   **阶段完成时**：更新 `state.json`：`phase → 3`

   **关键规则**：

- 不要偏离计划做额外的"改进"
- 遇到不确定的设计决策，先问用户

---

## Phase 4: Review

**进入条件**：编码基本完成。

**必做动作**：

1. 运行项目的质量门禁（格式化、lint、测试等，按项目 CLAUDE.md 中定义的命令执行）

2. Review 整个分支的 diff：`git diff main...HEAD`

3. 逐文件检查，关注：

   - Bug 和逻辑错误
   - 潜在的副作用（是否影响了非目标功能）
   - 安全问题
   - 遗漏的 TODO 或未完成项

4. 整理发现为问题清单（表格格式：严重程度 / 文件 / 描述）

5. 询问用户：「以上是 review 发现的问题，是否制定修复计划并执行？」

6. 用户确认后执行修复，修复完成后重新运行质量门禁

7. 质量门禁通过后，**提醒用户自行验证功能**：「代码 review 和质量门禁已通过，请你实际操作验证一下功能是否符合预期。验证后告诉我结果：」

   - **没问题** → 进入 Phase 5

   - **小问题**（UI 微调、文案修改、小 bug 等）→ 在当前 Phase 4 直接修复，修完后再次请用户验证

   - **大问题**（方向错误、需求理解偏差、需要重新设计）→ 回退到 Phase 1 重新制定需求，更新 `state.json`：`phase → 1`

     **阶段完成时**：更新 `state.json`：`phase → 4`

---

## Phase 5: 文档

**进入条件**：Phase 4 完成，用户验证功能符合预期。

**必做动作**：

1. 先确认：用户已在 Phase 4 末尾验证通过
2. 阅读项目已有文档，理解风格和结构模式
3. 回顾分支完整 diff：`git diff main...HEAD`
4. 生成两份文档：

### 文档 A：AI-facing 技术参考

放置在项目文档目录中，命名跟随已有风格。

**写作原则**：面向 AI 编码助手和开发者，用于后续代码修改时快速定位
**结构模板**（按需裁剪，没有的部分跳过）：

- Overview：一段话说清这个功能做什么

- Database Schema：SQL DDL 原文、表结构说明

- Backend API：路由表（Method / Path / Handler / Description）

- Key Types / Trait Methods：核心类型和方法签名

- Frontend Architecture：组件、事件协议、状态管理

- Key Flows：核心用户流程的步骤分解

- Files Changed Reference：改动文件清单表

  **风格要求**：

- 每个 API/方法/组件带**精确的文件路径和函数名**

- 用表格组织列表类信息，不用散文

- 贴入关键的 schema、类型签名、事件协议

- 信息密度优先，不需要背景故事

### 文档 B：人类可读总结 → `.vibe/`

放置在 `.vibe/` 目录，命名 `xxx-summary.md`。

**写作原则**：面向领导和团队成员，用于汇报和 review
**结构模板**：

- **功能概述**：一段话说清痛点和解决了什么（「此前...，现在...」）

- **功能详细说明**：按用户操作视角组织（不按代码模块），每个功能点用「场景 → 操作步骤 → 智能处理细节」结构

- **完成的工作量**：按前后端分开，具体数字（行数、文件数、接口数、模块数）

- **附带修复**：顺手修的 bug 和改进，单独列出

- **涉及文件**：分类统计表（新建 / 修改 / 删除）

  **风格要求**：

- 全中文

- 开头必须回答"之前不能做什么，现在能做什么"

- 工作量用数字说话，不用"做了很多"

- 如有前后对比数据（如重构前后的量化指标），用表格呈现

5. 更新 `state.json`：`phase → 5`，记录 `artifacts.summary`

---

## Phase 6: 提交

**进入条件**：两份文档已生成并经用户确认。

**必做动作**：

1. 向用户确认：「所有工作已完成，是否进入提交流程？」

2. 清理 `.vibe/` 中的临时文档（需求文档、计划文档），保留 summary

3. 提交并推送：

   ```bash
   git add <相关文件>
   git commit -m "feat: <简要描述>"
   git push -u origin <branch-name>
   ```

4. 更新 `state.json`：`phase → 6`

5. 提醒用户：「代码已推送到远端分支 `<branch-name>`，请前往仓库提交 MR。」

---

## 通用规则

1. **不要跳阶段**：每个阶段开始前必须向用户确认，不要自行从 Phase 1 跳到 Phase 3
2. **产出物是输入**：每个阶段的产出物是下一阶段的输入，确保产出物被妥善保存
3. **用户是决策者**：遇到不确定的设计选择，列出选项让用户决定，不要自行做重大决策
4. **状态必须同步**：每次阶段转换、产出物生成后，立即更新 `.vibe/state.json`
5. **上下文管理**：Phase 1 和 Phase 2 确认后，主动提示用户输入 `/compact` 压缩上下文，并说明"压缩后我会从 state.json 恢复状态"
6. **`.vibe/` 是临时工作区**：不应提交到版本库，Phase 0 时检查是否已在 `.gitignore`
